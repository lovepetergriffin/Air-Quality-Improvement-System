# NEP 环保公众监督系统答辩准备材料

## 1. 项目总体说明

### 1.1 项目定位

NEP 环保公众监督系统是一个面向空气污染反馈、任务指派、现场检测和数据统计分析的前后端分离项目。系统的核心目标是把公众发现污染、管理员分派处理、网格员现场确认、后台统计分析这几个环节串成完整业务闭环。

答辩时可以这样概括：

> 本项目是一个环保公众监督平台，公众监督员可以提交空气污染反馈，管理员后台负责查看反馈并指派网格员，网格员移动端负责接收任务、录入 SO2、CO、PM2.5 实测数据，系统最终生成确认 AQI 数据并在后台进行统计分析。

### 1.2 用户角色

系统中主要有三类用户：

| 用户角色 | 使用端 | 核心职责 |
| --- | --- | --- |
| 公众监督员 | `neps` 公众监督员移动端 | 注册、登录、选择省市地址、预估 AQI、提交污染反馈、查看个人历史反馈 |
| 网格员 | `nepg` 网格员移动端 | 登录、查看被指派的待处理任务、查看反馈详情、录入实测污染物数据、提交确认结果 |
| 管理员 | `nepm` 管理员后台端 | 登录后台、查看公众反馈、查看详情、指派网格员、查看确认 AQI 数据、查看统计分析 |

### 1.3 系统结构

最终项目是“三个前端 + 一个后端 + 一个数据库”的结构：

| 模块 | 目录 | 技术栈 | 默认端口 | 说明 |
| --- | --- | --- | --- | --- |
| 公众监督员端 | `项目源码/01.前端/neps` | Vue3 + Vite + Vue Router + Axios | `8081` | 移动端 H5，给公众监督员使用 |
| 网格员端 | `项目源码/01.前端/nepg` | Vue3 + Vite + Vue Router + Axios | `8083` | 移动端 H5，给网格员使用 |
| 管理员后台端 | `项目源码/01.前端/nepm` | Vue3 + Vite + Element Plus + Vuex + Axios + ECharts | `8082` | PC 后台管理系统 |
| 后端服务 | `项目源码/02.后端/nep-backend` | Spring Boot + MyBatis-Plus + MySQL | `8080` | 统一给三个前端提供接口 |
| 数据库 | `项目源码/03.数据库` | MySQL | - | 存储用户、反馈、AQI 标准、统计数据等 |

答辩时要强调：三个前端不是重复项目，而是面向不同角色的三个入口，共用同一个后端和同一套数据库。

### 1.4 端口关系

本地开发时端口划分如下：

```text
后端 Spring Boot： http://localhost:8080
公众监督员端 neps： http://localhost:8081
管理员后台端 nepm： http://localhost:8082
网格员端 nepg：   http://localhost:8083
```

之所以要分端口，是因为前后端分离开发时，每个服务都是独立启动的。前端通过 Axios 请求后端接口，后端通过 CORS 配置允许不同端口的前端访问。

### 1.5 完整业务闭环

系统完整流程如下：

```text
公众监督员注册/登录
-> 选择省份、城市、详细地址
-> 选择预估 AQI 等级并填写污染描述
-> 提交公众反馈
-> 管理员登录后台
-> 查看反馈列表和详情
-> 指派本地或异地网格员
-> 网格员登录移动端
-> 查看待处理任务
-> 查看任务详情并录入 SO2、CO、PM2.5
-> 系统计算确认 AQI 等级
-> 保存 Statistics 统计数据
-> 更新反馈状态为已处理
-> 管理员查看确认 AQI 数据和统计图表
```

这条链路是答辩时最重要的主线。老师问项目做了什么时，不要只说“写了登录、列表、查询”，要围绕这条业务闭环来讲。

## 3. 后端代码实现原理

### 3.1 后端整体分层

后端采用典型 Spring Boot 分层结构：

```text
Controller 层：接收前端 HTTP 请求，解析参数，返回 ResultVo
Service 层：处理业务逻辑，例如登录校验、状态更新、统计计算
Mapper 层：通过 MyBatis-Plus / MyBatis 访问数据库
POJO/DTO：承载数据库实体和前端请求参数
Utils：统一返回格式、工具方法
```

答辩说法：

> 后端按 Controller、Service、Mapper 三层划分。Controller 只负责接口入口和参数接收，Service 负责业务规则，Mapper 负责数据库操作。这样做可以降低耦合，方便后续维护和扩展。

### 3.2 Controller 层原理

Controller 是前端请求进入后端的入口。

主要 Controller：

| Controller | 职责 |
| --- | --- |
| `AdminController` | 管理员登录 |
| `SupervisorController` | 公众监督员登录、注册、查重、提交反馈、查询历史反馈 |
| `GridMemberController` | 网格员登录、查询网格员、查询任务 |
| `AqiFeedbackController` | 反馈列表、反馈详情、反馈更新、反馈指派、状态更新 |
| `StatisticsController` | 实测数据保存、确认 AQI 数据查询、统计分析 |
| `AqiController` | AQI 标准查询 |
| `GridProvinceController` | 省份查询 |
| `GridCityController` | 城市查询 |
| `CommonController` | 前端公共数据查询别名接口 |

常见注解：

| 注解 | 作用 |
| --- | --- |
| `@RestController` | 声明该类是 REST 接口控制器，返回 JSON 数据 |
| `@RequestMapping` | 定义接口公共路径 |
| `@GetMapping` | 处理 GET 请求，一般用于查询 |
| `@PostMapping` | 处理 POST 请求，一般用于新增、登录、复杂查询 |
| `@PutMapping` | 处理 PUT 请求，一般用于更新 |
| `@RequestBody` | 接收 JSON 请求体 |
| `@RequestParam` | 接收 URL 查询参数 |
| `@PathVariable` | 接收路径参数 |

示例解释：

```text
POST /supervisor/login
```

前端发送手机号和密码，Controller 用 `@RequestBody LoginDto` 接收，然后调用 `SupervisorService` 登录方法，最后返回统一 `ResultVo`。

### 3.3 Service 层原理

Service 是业务逻辑核心。

主要职责：

1. 登录校验：
   - 根据账号查询数据库。
   - 判断密码是否一致。
   - 登录成功返回用户对象，失败返回错误信息。

2. 反馈提交：
   - 接收公众端传来的省份、城市、地址、AQI 等级、反馈描述、手机号。
   - 构造 `AqiFeedback` 实体。
   - 设置初始状态为 `0`，表示未指派。
   - 保存到数据库。

3. 反馈指派：
   - 管理员选择网格员。
   - 后端更新反馈表中的 `gmId`、`assignDate`、`assignTime`、`state`。
   - 状态变为 `1`，表示已指派、待检测。

4. 网格员确认：
   - 网格员提交 SO2、CO、PM2.5 实测值。
   - 后端保存 `Statistics` 统计记录。
   - 同时把原反馈状态更新为 `2`，表示已处理。

5. 统计分析：
   - 按省市聚合。
   - 按 AQI 等级分布聚合。
   - 按时间趋势聚合。
   - 返回给管理员端图表展示。

答辩说法：

> Service 层主要保证业务状态流转正确，例如公众提交反馈后是未指派，管理员指派后是待检测，网格员提交实测数据后变为已处理。状态变化不放在前端控制，而是以后端保存结果为准。

### 3.4 Mapper / MyBatis-Plus 原理

Mapper 负责数据库访问。

项目中主要有两类查询：

1. 简单查询：
   - 按 ID 查询。
   - 按手机号查询。
   - 查询所有省份、城市、AQI 标准。

2. 复杂查询：
   - 反馈分页查询。
   - 反馈详情多表关联。
   - Statistics 详情多表关联。
   - 省份污染统计。
   - AQI 分布统计。
   - AQI 趋势统计。

分页查询思路：

```text
pageNum：当前第几页
pageSize：每页多少条
offset = (pageNum - 1) * pageSize
SQL 使用 LIMIT offset, pageSize 查询当前页数据
再用 COUNT 查询总条数
最后封装为 PageResult 返回给前端
```

聚合统计思路：

```text
使用 GROUP BY 按省份、城市、AQI 等级或日期分组
使用 COUNT、AVG、MAX 等聚合函数计算数量、平均值、最大值
返回统计 DTO 或 Map 给前端
前端用表格或 ECharts 展示
```

答辩说法：

> 普通列表可以用简单查询，但统计分析必须使用聚合 SQL。比如 AQI 分布统计是按 AQI 等级分组后计算数量，趋势统计是按日期分组后计算每天或每月的 AQI 情况。

### 3.5 ResultVo 统一响应

后端统一返回 `ResultVo`，包含：

| 字段 | 含义 |
| --- | --- |
| `flag` | 请求是否成功 |
| `code` | 状态码，成功一般是 `200` |
| `message` / `msg` | 提示信息 |
| `data` | 实际返回数据 |

统一响应的好处：

- 前端判断成功失败更简单。
- 错误提示格式统一。
- 不同接口返回结构一致，便于维护。

答辩说法：

> 我们没有让每个接口随意返回不同格式，而是统一封装成 ResultVo。这样前端只需要统一判断 `code` 或 `flag`，然后从 `data` 中取业务数据。

### 3.6 DTO 和 POJO 的作用

POJO 一般对应数据库表，例如：

| POJO | 对应含义 |
| --- | --- |
| `Admin` | 管理员 |
| `Supervisor` | 公众监督员 |
| `GridMember` | 网格员 |
| `AqiFeedback` | 公众反馈 |
| `Statistics` | 网格员确认后的统计数据 |
| `GridProvince` | 省份 |
| `GridCity` | 城市 |
| `Aqi` | AQI 标准 |

DTO 一般用于接收前端请求参数，例如：

| DTO | 作用 |
| --- | --- |
| `LoginDto` | 登录参数，兼容管理员、网格员、公众监督员登录字段 |
| `FeedbackCreateDto` | 公众提交反馈 |
| `FeedbackUpdateDto` | 更新反馈 |
| `AssignFeedbackDto` | 管理员指派网格员 |
| `ConfirmAqiDto` | 网格员提交实测数据 |
| `FeedbackQueryDto` | 反馈查询条件 |
| `PageResult` | 分页返回结果 |

为什么不用一个实体接收所有请求：

> 数据库实体表示表结构，请求 DTO 表示前端传参。二者分开可以避免前端传入不该修改的字段，也让接口语义更清晰。

### 3.7 反馈状态流转

反馈状态是业务闭环的关键。

| 状态值 | 含义 | 产生场景 |
| --- | --- | --- |
| `0` | 未指派 | 公众监督员刚提交反馈 |
| `1` | 已指派 / 待检测 | 管理员已经指派给网格员 |
| `2` | 已处理 / 已确认 | 网格员已经录入实测数据并提交 |

状态流转：

```text
公众提交反馈：state = 0
管理员指派网格员：state = 1
网格员提交实测数据：state = 2
```

答辩时如果老师问“为什么网格员端任务做完就没有了”，可以回答：

> 网格员端只查询当前网格员状态为 `1` 的待处理任务。任务提交后状态变为 `2`，已经不是待处理任务，所以列表显示 No Data 是正常业务结果。

## 4. 前端代码实现原理

### 4.1 三个前端为什么分开

三个前端面向不同用户角色，业务场景、页面布局和操作方式不同。

| 前端 | 为什么独立 |
| --- | --- |
| `neps` | 面向公众监督员，移动端表单和反馈流程简单，重点是提交和查看自己的反馈 |
| `nepg` | 面向网格员，移动端任务处理流程，重点是查看任务和录入实测数据 |
| `nepm` | 面向管理员，PC 端后台系统，重点是表格、分页、指派、统计图表 |

答辩说法：

> 三个前端不是重复开发，而是按角色拆分。公众端和网格员端更适合手机使用，管理员端需要大量表格和统计图表，所以做成 PC 后台。

### 4.2 Vue3 基本原理

三个前端都使用 Vue3。

核心知识点：

| 知识点 | 项目中的使用 |
| --- | --- |
| `ref` | 定义响应式基本数据、数组，如任务列表、AQI 列表、输入值 |
| `reactive` | 定义表单对象，如登录表单、查询表单 |
| `computed` | 根据输入值实时计算 AQI 等级 |
| `provide/inject` | 在 `App.vue` 中注入 Axios，在页面组件中取出使用 |
| `v-model` | 表单双向绑定 |
| `v-for` | 渲染列表，如反馈列表、AQI 列表、省市列表 |
| `v-if` | 条件展示，如 No Data、切换本地/异地指派 |
| `:style` | 动态绑定 AQI 等级颜色 |
| `@click` | 绑定按钮点击事件 |
| `scoped` | 限制组件样式只在当前页面生效 |

答辩说法：

> Vue 的响应式机制让页面数据和变量保持同步。比如用户输入污染物数值后，`computed` 自动重新计算 AQI 等级，页面上的颜色和等级会实时更新。

### 4.3 Axios 调接口

三个前端都通过 Axios 请求后端接口。

基本方式：

```text
App.vue 中配置 axios.defaults.baseURL
页面组件通过 inject('axios') 获取 Axios
调用 axios.get 或 axios.post 请求后端接口
根据 response.data.code 或 response.data.flag 判断是否成功
从 response.data.data 中取业务数据
```

当前项目做了局域网适配：

```text
const apiHost = window.location.hostname || 'localhost';
axios.defaults.baseURL = `http://${apiHost}:8080/`;
```

这意味着：

- 本机打开 `localhost:8081` 时，请求 `localhost:8080`。
- 手机打开 `10.17.112.79:8081` 时，请求 `10.17.112.79:8080`。

答辩说法：

> 这样处理后，别人通过局域网 IP 访问我的前端时，前端不会再请求他自己手机上的 localhost，而是请求提供页面的那台电脑的 8080 后端。

### 4.5 sessionStorage 登录态

登录成功后，前端会把当前用户信息保存到 `sessionStorage`。

| 前端 | 保存内容 |
| --- | --- |
| `neps` | `supervisor` |
| `nepg` | `gridMember` |
| `nepm` | `admins` |

为什么用 `sessionStorage`：

- 浏览器标签关闭后自动清除。
- 适合保存当前会话登录状态。
- 比一直存在的 `localStorage` 更适合临时登录信息。

答辩说法：

> 登录态只需要在当前浏览器会话中有效，所以用 sessionStorage。它比 localStorage 更符合“关闭浏览器后登录态消失”的需求。

## 5. 核心业务场景解释

### 5.1 公众提交反馈

操作路径：

```text
公众端登录
-> 选择省市地址
-> 选择 AQI 等级
-> 填写污染描述
-> 提交
```

数据流：

```text
neps 前端
-> POST /supervisor/submitFeedback
-> SupervisorController
-> AqiFeedbackService
-> AqiFeedbackMapper
-> aqi_feedback 表插入记录
```

关键字段：

| 字段 | 含义 |
| --- | --- |
| `telId` | 公众监督员手机号 |
| `provinceId` | 省份 ID |
| `cityId` | 城市 ID |
| `address` | 详细地址 |
| `estimatedGrade` / `aqiId` | 公众预估 AQI 等级 |
| `information` | 污染描述 |
| `state` | 初始状态，值为 `0` |

答辩回答：

> 公众提交反馈时，前端把省市、地址、AQI 等级、描述和当前登录手机号传给后端，后端保存到反馈表，并把状态设置为未指派。这样管理员后台就可以看到新反馈。

### 5.2 管理员指派网格员

操作路径：

```text
管理员登录
-> 进入反馈列表
-> 查看反馈详情
-> 点击指派
-> 选择本地或异地网格员
-> 确认指派
```

本地指派：

```text
根据反馈所在省市查询当地网格员
管理员选择一个网格员
保存 gmId 到反馈记录
```

异地指派：

```text
管理员手动选择省份
根据省份加载城市
根据城市加载网格员
选择网格员后保存 gmId
```

指派后的变化：

```text
aqi_feedback.gm_id = 被指派网格员 ID
aqi_feedback.assign_date = 当前日期
aqi_feedback.assign_time = 当前时间
aqi_feedback.state = 1
```

答辩回答：

> 指派本质上是更新反馈记录，把网格员 ID、指派日期、指派时间写入反馈表，并把状态改成已指派。网格员端就是根据当前登录网格员 ID 查询状态为已指派的任务。

### 5.3 网格员处理任务

操作路径：

```text
网格员登录
-> 查看任务列表
-> 点击去检测
-> 查看任务详情
-> 输入 SO2、CO、PM2.5
-> 系统计算 AQI
-> 提交确认数据
```

数据流：

```text
nepg 前端
-> GET /aqiFeedback/listAssignedTasks?gmId=...
-> 查询当前网格员待处理任务

nepg 前端
-> GET /aqiFeedback/detail/{afId}
-> 查询反馈详情

nepg 前端
-> POST /statistics/save
-> 保存实测统计数据
-> 更新反馈状态为 2
```

答辩回答：

> 网格员端不是查询所有反馈，而是只查询分配给当前网格员且未处理的任务。录入实测数据后，系统保存 Statistics 记录，并把对应反馈状态更新为已处理。

### 5.4 管理员查看统计分析

统计来源：

```text
公众提交的是 AqiFeedback
网格员确认后生成 Statistics
统计分析主要基于 Statistics 表
```

统计页面：

| 页面 | 统计内容 |
| --- | --- |
| 省份污染统计 | 按省份、城市统计检测数量和污染指标 |

| AQI 趋势统计 | 按日期或月份统计 AQI 变化 |
| 其他统计 | AQI 总数、良好数量、省份覆盖、城市覆盖等 |

答辩回答：

> 管理端统计不是前端自己算的，而是后端通过 SQL 聚合后返回结果，前端负责表格和图表展示。这样数据口径统一，也避免前端处理大量原始数据。

## 6. 重要知识点整理

### 6.1 Spring Boot

重点掌握：

- Spring Boot 用于快速搭建后端服务。
- Controller 暴露 REST API。
- Service 组织业务逻辑。
- Mapper 访问数据库。
- 配置文件 `application.yml` 管理端口、数据库连接等配置。

可答辩说法：

> Spring Boot 简化了 Java Web 项目配置，项目通过注解就可以暴露接口，配合 MyBatis-Plus 完成数据库访问。

### 6.2 MyBatis-Plus / MyBatis

重点掌握：

- MyBatis-Plus 适合基础 CRUD。
- MyBatis XML 适合复杂 SQL，比如多表关联和聚合统计。
- 查询详情时经常需要关联省份、城市、AQI、监督员、网格员等表。

可答辩说法：

> 简单增删改查可以交给 MyBatis-Plus，复杂统计和多表查询用 SQL 更直观，例如 AQI 分布统计和省份污染统计都需要 GROUP BY。

### 6.3 Vue3 响应式

重点掌握：

- `ref` 包装基本类型、数组。
- `reactive` 包装对象。
- `computed` 用于由已有状态推导新结果。
- 模板会随着响应式数据变化自动更新。

可答辩说法：

> Vue3 的响应式机制让数据变化自动反映到页面。网格员录入污染物数值后，计算属性会自动重新计算 AQI 等级，不需要手动操作 DOM。

### 6.4 Vue Router

重点掌握：

- 路由表把 URL 和组件对应起来。
- `router.push` 用于页面跳转。
- `useRoute` 用于读取当前路由参数。
- 路由守卫用于登录拦截。

可答辩说法：

> 路由守卫在进入业务页面前检查登录态，如果没有登录信息就跳回登录页，这是前端权限控制的一种基础方式。

### 6.5 Axios

重点掌握：

- Axios 用于前后端 HTTP 通信。
- `baseURL` 统一配置后端地址。
- GET 用于简单查询，POST 用于登录、提交、复杂查询。
- 前端统一从 `response.data.data` 取业务数据。

可答辩说法：

> 前端不直接访问数据库，而是通过 Axios 请求后端接口。后端负责数据查询和业务逻辑，前端只负责展示和交互。

### 6.6 Element Plus

重点掌握：

- 主要用于管理员后台端。
- 适合快速实现后台布局、菜单、表格、分页、表单和提示框。

可答辩说法：

> 管理员端是 PC 后台，页面以表格、查询和统计为主，用 Element Plus 可以提高开发效率，并保持后台页面风格统一。

### 6.7 ECharts

重点掌握：

- 用于把统计数据可视化。
- AQI 分布适合柱状图或饼图。
- AQI 趋势适合折线图。
- 图表数据来源于后端统计接口。

可答辩说法：

> ECharts 只负责展示，真正的统计结果由后端 SQL 聚合得出。前端拿到统计数组后转成图表配置项显示。

### 6.8 前后端分离

重点掌握：

- 前端和后端独立启动。
- 前端通过 HTTP 接口获取数据。
- 后端返回 JSON。
- 前后端通过接口字段约定协作。

可答辩说法：

> 前后端分离的好处是职责清晰。后端只提供接口，前端只做页面和交互。三个前端都能复用同一个后端，减少重复开发。

### 6.9 局域网访问

重点掌握：

- `localhost` 只代表当前设备。
- 手机访问电脑项目时，不能用 `localhost`，要用电脑局域网 IP。
- 前端页面访问 `10.17.112.79:8081`，接口应请求 `10.17.112.79:8080`。

可答辩说法：

> 局域网访问时，手机上的 localhost 指的是手机自己，所以项目把接口地址改成根据当前页面 host 自动拼接后端地址，这样别人访问我的电脑 IP 时，接口也会请求我的电脑 IP。

## 7. 答辩常见问题与参考回答

### 7.1 项目架构类

#### Q1：你的项目整体是怎么设计的？

参考回答：

> 项目采用三前端一后端结构。`neps` 是公众监督员移动端，负责提交污染反馈；`nepg` 是网格员移动端，负责处理管理员指派的检测任务；`nepm` 是管理员 PC 后台，负责反馈管理、指派和统计分析；`nep-backend` 是统一 Spring Boot 后端，三个前端都通过 Axios 请求这个后端，后端再访问 MySQL 数据库。

追问回答：

> 这样划分是因为三类用户操作场景不同。公众和网格员主要在移动端操作，管理员需要表格、分页和统计图表，所以做成 PC 后台。

#### Q2：为什么要做三个前端，而不是一个前端？

参考回答：

> 三个角色的界面和操作差异较大。公众监督员只需要提交和查看自己的反馈，网格员只需要处理自己的任务，管理员需要管理所有反馈和统计数据。如果放在一个前端里，路由和权限会更复杂，移动端和 PC 端布局也不好统一，所以按角色拆成三个前端更清晰。

容易答错点：

> 不要回答“因为视频里就是三个”。应该从角色、场景、布局和权限角度解释。

#### Q3：三个前端怎么共用后端？

参考回答：

> 三个前端都通过 Axios 请求同一个 Spring Boot 后端。后端暴露统一接口，例如登录、反馈、指派、统计等接口。不同前端调用自己需要的接口，数据库也是同一套，所以数据可以互通。

#### Q4：这个项目的业务闭环是什么？

参考回答：

> 公众监督员提交污染反馈，管理员后台查看反馈并指派网格员，网格员移动端看到任务后录入实测数据，系统保存确认 AQI 数据并更新反馈状态，最后管理员可以在后台查看统计分析。这就是从反馈到处理再到统计的闭环。

### 7.2 数据库和状态类

#### Q5：反馈表和统计表有什么区别？

参考回答：

> 反馈表 `aqi_feedback` 存的是公众提交的原始反馈，包括省市、地址、污染描述、预估 AQI 和处理状态。统计表 `statistics` 存的是网格员实地检测后的确认数据，包括 SO2、CO、PM2.5、确认 AQI、确认时间和网格员信息。前者是公众上报，后者是现场确认结果。

#### Q6：反馈状态是怎么流转的？

参考回答：

> 公众提交反馈后状态是 `0`，表示未指派。管理员指派网格员后状态变为 `1`，表示已指派、待检测。网格员提交实测数据后状态变为 `2`，表示已处理或已确认。网格员端只查询状态为 `1` 的任务。

#### Q7：为什么网格员任务完成后列表显示 No Data？

参考回答：

> 因为网格员端显示的是待处理任务。任务完成后状态从 `1` 变为 `2`，已经不是待处理任务，所以不会继续显示。如果该网格员没有其他待处理任务，列表就会显示 No Data。

#### Q8：网格员是不是只能有一个任务？

参考回答：

> 不是。网格员可以被指派多个任务。当前看到一个任务或没有任务，只是数据库当前数据状态决定的。只要管理员继续给这个网格员指派反馈，列表就会显示多条待处理任务。

### 7.3 后端接口类

#### Q9：后端为什么要分 Controller、Service、Mapper？

参考回答：

> 这是为了分层解耦。Controller 负责接收前端请求和返回结果，Service 负责业务逻辑，比如登录校验、状态流转、指派规则，Mapper 负责数据库操作。分层后代码职责清晰，后续维护也方便。

#### Q10：ResultVo 有什么作用？

参考回答：

> `ResultVo` 是统一响应格式，里面包含成功标志、状态码、提示信息和业务数据。这样前端不用针对每个接口写不同判断，只要统一看 `code`、`flag` 和 `data` 即可。

#### Q11：分页查询怎么实现？

参考回答：

> 前端传 `pageNum` 和 `pageSize`，后端计算偏移量 `offset = (pageNum - 1) * pageSize`，SQL 使用 `LIMIT offset, pageSize` 查询当前页数据，再用 `COUNT` 查询总记录数，最后封装成 `PageResult` 返回给前端。

#### Q12：为什么详情查询要多表关联？

参考回答：

> 反馈表里通常只保存 `provinceId`、`cityId`、`aqiId`、`telId`、`gmId` 这些外键或编号。前端展示时需要省份名称、城市名称、AQI 名称、监督员姓名、网格员姓名，所以后端要关联省份表、城市表、AQI 表、监督员表和网格员表，返回完整展示数据。

#### Q13：为什么有些接口有多个路径别名？

参考回答：

> 因为当前完整项目需要兼容视频课件中的接口路径和后来整合后的前端路径。为了不破坏已有页面，后端保留了课件路径，同时补充了简化路径。这样三端联调更稳定。

### 7.4 前端实现类

#### Q14：前端如何和后端通信？

参考回答：

> 前端使用 Axios 发送 HTTP 请求。每个前端在 `App.vue` 中统一配置后端 `baseURL`，页面组件通过 `inject('axios')` 获取 Axios，然后调用 `get` 或 `post` 请求后端接口，后端返回 JSON 后前端更新页面数据。

#### Q15：为什么使用 provide/inject？

参考回答：

> Axios 是很多页面都要用的公共对象。如果每个页面都单独 import 和配置，会重复。我们在 `App.vue` 中 `provide` 一次，然后页面通过 `inject` 获取，方便统一管理。

#### Q16：sessionStorage 存什么？为什么不用 localStorage？

参考回答：

> 登录成功后，前端把当前用户信息存到 sessionStorage，例如公众端存 `supervisor`，网格员端存 `gridMember`，管理员端存 `admins`。用 sessionStorage 是因为登录态只需要当前会话有效，关闭浏览器后自动清除，比 localStorage 更适合临时登录信息。

#### Q17：路由守卫的作用是什么？

参考回答：

> 路由守卫在页面跳转前执行，用来检查用户是否登录。如果用户没有登录却访问业务页面，前端会把他重定向到登录页，避免直接通过地址栏进入系统页面。

#### Q18：Vue3 的 computed 在项目哪里用了？

参考回答：

> 网格员端任务详情页用了 `computed`。网格员输入 SO2、CO、PM2.5 后，系统会根据 AQI 标准实时计算每一项对应的等级，再取三个等级中最严重的作为综合 AQI。输入值变化时，computed 会自动重新计算，页面上的等级和颜色也会更新。

#### Q19：为什么移动端页面大量使用 vw？

参考回答：

> `vw` 是相对于视口宽度的单位，适合移动端适配。不同手机屏幕宽度不同，用 `vw` 可以让按钮、表单、字体按屏幕比例缩放。

### 7.5 AQI 和统计类

#### Q20：AQI 等级是怎么计算的？

参考回答：

> 系统先从后端获取 AQI 标准，每个等级包含 SO2、CO、PM2.5 的区间。网格员输入三个实测值后，前端分别判断每个值落在哪个等级区间，再取三项中最严重的等级作为综合 AQI。

#### Q21：为什么综合 AQI 取最严重的一项？

参考回答：

> 因为空气质量评价中，只要某一项污染物达到较严重等级，整体空气质量就应该按较严重等级处理。这样更符合环保监测的安全原则。

#### Q22：统计图表的数据从哪里来？

参考回答：

> 图表数据来自后端统计接口，不是前端自己编的。后端根据 Statistics 表做 SQL 聚合，比如按 AQI 等级分组统计数量，按日期分组统计趋势，前端拿到结果后用 ECharts 展示。

#### Q23：AQI 分布统计怎么做？

参考回答：

> 后端按 AQI 等级进行 `GROUP BY`，统计每个等级对应的记录数量，返回给前端。前端把这个数组转换成 ECharts 所需的数据格式，展示为柱状图或饼图。

#### Q24：AQI 趋势统计怎么做？

参考回答：

> 后端按确认日期或月份分组，统计不同时间段的 AQI 情况，例如平均等级、最高等级或记录数量，然后前端用折线图展示变化趋势。

#### Q25：省份污染统计怎么做？

参考回答：

> 后端根据 Statistics 表关联省份表和城市表，按省市分组，统计检测数量以及 SO2、CO、PM2.5 等指标。这样管理员可以看到不同地区的污染数据分布。

### 7.6 管理端功能类

#### Q26：管理员指派网格员时，本地指派和异地指派有什么区别？

参考回答：

> 本地指派是根据反馈所在省市查询当地网格员；异地指派是管理员手动选择其他省市和网格员。二者最终都是把选中的网格员 ID 保存到反馈记录里，并把反馈状态改为已指派。

#### Q27：管理员端为什么用 Element Plus？

参考回答：

> 管理端是 PC 后台，主要是表格、分页、查询表单、菜单和详情展示。Element Plus 提供了成熟的后台组件，可以提高开发效率，也让页面风格统一。

#### Q28：管理员端为什么要用 Vuex？

参考回答：

> 当前项目中 Vuex 主要用于保存全局配置，例如每页显示数量。这样多个列表页面可以统一读取同一个配置，避免每个页面重复写固定值。

### 7.7 运行和部署类

#### Q29：为什么别人访问你的电脑要用 IP，而不是 localhost？

参考回答：

> `localhost` 永远表示当前设备。如果别人手机访问 `localhost:8081`，访问的是他自己的手机，不是我的电脑。局域网访问必须使用我电脑的 IPv4 地址，例如 `10.17.112.79:8081`。

#### Q30：页面能打开，但提示无法连接后端，可能是什么原因？

参考回答：

> 说明前端服务能访问，但后端接口访问失败。常见原因是后端 `8080` 没启动、防火墙拦截、前端请求了错误地址、数据库连接失败或后端报 500。排查时先访问 `http://电脑IP:8080/aqi/listAqiAll` 看后端是否能返回 JSON。

#### Q31：8080 被占用怎么办？

参考回答：

> 可以用 `netstat -ano | findstr :8080` 查看占用端口的进程 PID，再用 `taskkill /PID 进程号 /F` 结束旧进程，然后重新启动后端。

#### Q32：如果部署到云服务器，和本机运行有什么区别？

参考回答：

> 本机运行依赖我的电脑开机和本地网络，云服务器部署后别人可以通过公网 IP 或域名访问，不依赖我的电脑。Docker 则是解决运行环境一致性的问题，最规范的方式是云服务器加 Docker Compose，再用 Nginx 统一代理前端和后端。

### 7.8 项目改动和视频一致性类

#### Q33：你的项目和视频内容完全一样吗？

参考回答：

> 核心业务逻辑和视频一致，包括登录、反馈提交、分页、指派、网格员确认、统计分析等。当前项目在视频基础上做了整合和补全，把 NEPS、NEPG、NEPM 三个前端统一接到一个后端，并补充了部分兼容接口，方便完整联调。

#### Q34：为什么后端接口路径和课件有些不完全一样？

参考回答：

> 为了兼容不同阶段的前端代码，后端保留了课件中的路径，也补充了一些简化路径或别名。这样既能对应视频内容，也能保证最终三端项目正常运行。

#### Q35：你觉得项目最核心的技术点是什么？

参考回答：

> 最核心的是前后端分离下的完整业务闭环和状态流转。前端负责不同角色页面，后端统一处理登录、反馈、指派、统计等业务，数据库保存状态。除此之外，网格员端的 AQI 实时计算和后台统计分析也是比较重要的技术点。

## 9. 易错点提醒

### 9.1 不要把三个前端说成三个后端

正确说法：

> 项目有三个前端，一个后端。

错误说法：

> 项目有三个系统后端。

### 9.2 不要说 AQI 是随便选的

正确说法：

> 公众端提交的是预估 AQI，网格员端根据 SO2、CO、PM2.5 实测值确认 AQI。

### 9.3 不要说 No Data 是报错

正确说法：

> No Data 表示当前网格员没有待处理任务，可能是未被指派，也可能是任务已经完成。

### 9.4 不要说前端直接操作数据库

正确说法：

> 前端通过 Axios 请求后端接口，后端通过 Mapper 操作数据库。

### 9.5 不要说 sessionStorage 是真正安全认证

正确说法：

> sessionStorage 是前端保存登录态和做页面拦截的方式，真正严格的权限控制应该以后端鉴权为准。本项目实训阶段主要实现基础登录拦截。

### 9.6 不要把 Statistics 和 AqiFeedback 混淆

正确说法：

> AqiFeedback 是公众反馈，Statistics 是网格员确认后的实测统计数据。
